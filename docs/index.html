March Madness Upset Analytics

Built an end-to-end analytics platform to identify and explain potential NCAA Tournament upsets. The app combines historical performance data, team-level features, and probabilistic modeling to surface high-risk matchups and generate data-driven bracket recommendations through an interactive Streamlit dashboard.

Python pandas scikit-learn Streamlit Plotly NumPy
More details

Problem

March Madness is inherently unpredictable, making it difficult to consistently identify which lower-seeded teams have a realistic chance to win. Most brackets rely on intuition or surface-level stats, which fail to capture deeper performance signals. There is a need for a structured, data-driven approach that quantifies upset probability and provides clear reasoning behind each prediction.

Approach

Built a full data pipeline that integrates NCAA tournament data, team-level performance metrics, and seed information to construct Round 1 matchups and beyond. Engineered features such as efficiency metrics, power ratings, and historical performance indicators, then applied probabilistic modeling to estimate win likelihoods for each game. Developed logic to identify underdogs, assign upset probabilities, and generate interpretable “why” factors based on feature differentials. Deployed the system as an interactive Streamlit app with modules for upset alerts, bracket building, and tournament simulations.

Results & Impact

Delivered an interactive Streamlit app capable of generating upset probability scores for any Round 1 matchup. The model surfaces high-risk games with interpretable reasoning — showing which specific feature differentials (e.g. efficiency gap, power rating delta) drive each prediction. Bracket simulation module allows users to run full tournament scenarios and compare upset-adjusted outcomes against seed-based expectations.

NBA Player Clustering by Playing Style

Clustered 301 NBA players from the 2023–24 regular season into six distinct playing-style archetypes using K-Means, PCA, and UMAP. The project spans a full ML pipeline — data collection via the NBA API, feature engineering, unsupervised modeling, and 13 interactive visualizations — to answer the question: what types of players actually exist in the modern NBA?

Python nba_api scikit-learn pandas NumPy UMAP Plotly seaborn Jupyter Notebooks
More details

Problem

NBA player positions (Guard, Forward, Center) are too coarse to capture how modern players actually play. A stretch-four and a rim-running big both count as "power forwards," yet they fill completely different roles. The goal was to let the data define player types from scratch — without relying on traditional labels — using only per-game statistics and engineered performance features.

Approach

Pulled per-game stats for 572 players via the NBA API and filtered to 301 qualified players (≥30 games, ≥15 minutes per game). Engineered 13 features including per-minute scoring and rebounding rates, three-point rate, paint score, assist ratio, and an encoded position score. Applied StandardScaler before running K-Means with elbow-method and silhouette-score tuning to select K=6. Reduced dimensions with PCA and UMAP for visualization, then used an auction-based labeling algorithm to assign each cluster a unique, meaningful archetype name based on the centroid feature profile.

Results & Impact

Produced six interpretable archetypes — Elite Scorer, Playmaker, 3PT Specialist, Rebounding Big, Rim Protector, and Perimeter Defender — with well-known players anchoring each group (e.g., Luka Dončić in Elite Scorer, Rudy Gobert in Rebounding Big). Interactive Plotly scatter plots and radar charts allow exploration of all 301 players and their cluster profiles. The clustering confirmed that modern NBA roles do not map cleanly onto traditional positions, and that three-point tendency and playmaking together drive the largest variance across players.

NYC 311 Cloud Analytics (AWS)

Built a cloud-based analytics pipeline on AWS to explore 200,000 NYC 311 service requests from Q1 2026. Ingested raw data into S3, queried it with Athena using SQL, and produced stakeholder-ready findings on complaint volume by borough, agency, and complaint type — all without provisioning a single server.

AWS S3 AWS Athena SQL Python pandas
More details

Problem

New York City receives millions of 311 service requests each year across dozens of agencies. Understanding which boroughs generate the most complaints, which agencies handle the highest volume, and how complaint type varies by location requires working with large datasets efficiently. The challenge was to stand up a scalable, serverless query environment from raw CSV data without managing infrastructure.

Approach

Loaded a 200k-row sample of NYC 311 complaints (Jan–Mar 2026) and a companion agencies lookup table into an S3 bucket. Registered both tables in AWS Athena and wrote SQL queries to count complaints by agency, join agency names, group by borough and complaint type, and identify the top complaint categories per area. All queries run serverlessly against data stored in S3, making the workflow reproducible and cost-effective.

Results & Impact

Identified the top 10 agencies by complaint volume and mapped complaint types — including Snow or Ice, Noise, and HEAT/HOT WATER — across all five boroughs. The Athena-based setup means any stakeholder with AWS access can run the same queries against updated data without additional engineering. This project demonstrates a practical cloud data workflow: raw files in S3, schema-on-read querying with Athena, and shareable SQL reports.
